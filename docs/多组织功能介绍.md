# 多组织多部门功能介绍

## 1. 功能概述

本次多组织功能是在现有 NoDeskClaw 基础上，为 toB 场景补齐“多个公司并行使用、组织内多部门分级管理、办公室按部门范围控人”的能力。

核心目标有三点：

- 支持多个组织（Organization，组织）并行隔离使用
- 支持组织内树形部门（Department，部门）管理人员归属
- 支持办公室（Workspace，办公室）按部门范围控制可见成员、可添加成员和可放置的人类工位

这套能力主要服务于以下场景：

- SaaS 平台向多个不同企业开放
- 同一企业内部存在总部、事业部、子部门等层级结构
- 不同部门负责人只管理自己部门树内的成员
- 办公室只允许特定部门的成员加入和协作

## 3. 数据模型设计

### 3.1 Organization（组织）

组织仍然是企业级隔离边界。

- 不同组织之间的数据天然隔离
- 用户通过 `OrgMembership`（组织成员关系）进入组织
- 组织管理员拥有组织级管理权限

### 3.2 Department（部门）

新增 `Department`（部门）模型，用于表达组织内树形结构。

关键字段：

- `org_id`：所属组织
- `parent_id`：父部门
- `name`：部门名称
- `slug`：部门唯一标识
- `description`：部门说明
- `sort_order`：排序
- `is_active`：是否启用

设计特点：

- 一个组织下可以有多个一级部门
- 每个部门可以有多个子部门
- 通过 `parent_id` 实现树形结构

### 3.3 DepartmentMembership（部门成员关系）

新增 `DepartmentMembership`（部门成员关系）模型，用于表达“用户属于哪个部门”。

关键字段：

- `org_id`：所属组织
- `department_id`：所属部门
- `user_id`：所属用户
- `role`：部门内角色，支持 `member` / `manager`
- `is_primary`：是否主部门

设计特点：

- 一个用户必须有且仅有一个主部门
- 一个用户可以加入多个兼任部门
- 部门负责人通过 `role = manager` 表示

### 3.4 Workspace（办公室）范围配置

在 `Workspace`（办公室）模型上增加了部门联动字段：

- `visibility_scope`：可见范围
- `allowed_department_ids`：允许访问的部门根节点
- `auto_sync_mode`：自动同步模式

当前默认行为：

- 老办公室默认仍是 `org`
- 开启部门范围后，可按部门树做成员过滤

## 4. 权限模型

本次功能采用“组织权限 + 部门权限 + 办公室权限”三层叠加。

### 4.1 平台超管

- 管理所有组织
- 查看和操作全部组织数据

### 4.2 组织管理员

- 管理本组织的部门树
- 管理本组织成员
- 管理本组织下办公室配置和成员

### 4.3 部门管理员

- 只能管理“自己所在部门及其所有子部门”的成员
- 可以在受控范围内搜索组织成员
- 可以在受控范围内把成员加入办公室
- 可以在受控范围内放置人类工位

### 4.4 办公室成员权限

办公室仍保留原有 `WorkspaceMember.permissions`（办公室成员权限）模型。

也就是说：

- 部门只决定“能管谁”
- 办公室权限决定“在办公室里能做什么”

## 5. 实现原理

### 5.1 多组织隔离

所有核心查询都以 `org_id` 为边界。

例如：

- 列部门时按组织过滤
- 列组织成员时按组织过滤
- 办公室搜索成员时，先定位办公室所属组织，再在该组织内筛选成员

这保证了 A 组织不能看到 B 组织的数据。

### 5.2 树形部门作用域

后端通过 `department_service` 统一处理部门树能力：

- 查询某部门的全部子部门
- 查询某个用户可管理的部门树范围
- 查询用户主部门、兼任部门和是否为部门管理员

这样做的好处是：

- 权限判断逻辑集中，不会散落在各 API
- 前端只消费结果，不自己拼部门树权限

### 5.3 组织成员与部门归属联动

组织成员页现在支持维护：

- 主部门
- 兼任部门
- 是否部门管理员

成员接口返回的数据里也会补充：

- `primary_department_name`
- `secondary_departments`
- `is_department_manager`

这样前端成员列表可以直接展示部门归属。

### 5.4 办公室范围过滤

办公室范围控制主要落在两个地方：

- `workspace_member_service`：成员搜索、权限检查
- `workspace_service`：加办公室成员、列办公室成员

当前实现目标是：

- 办公室限制为某些部门树后，只能操作这些部门树内成员
- 不在允许范围内的成员，不能被加入办公室，也不能放置人类工位

### 5.5 人类工位联动

首页“放置人类工位”的成员候选来源已经从“办公室成员”扩展为“办公室成员 + 可加入办公室的组织成员”。

链路如下：

1. 搜索当前可加入办公室的成员
2. 如果成员尚未加入办公室，先调用加成员接口
3. 成员加入成功后，再调用创建人类工位接口

这使得普通组织成员可以直接放置为人类工位，不需要先手工去办公室成员页加一次。

## 6. Portal 功能说明

### 6.1 部门管理页

页面位置：

- 组织设置 -> 部门管理

支持能力：

- 创建部门
- 编辑部门
- 删除部门
- 查看部门树
- 查看部门成员

### 6.2 组织成员页

支持能力：

- 查看成员主部门
- 查看成员兼任部门
- 按部门筛选成员
- 直接创建成员并指定部门
- 修改成员部门归属

### 6.3 办公室设置页

支持能力：

- 配置办公室可见范围
- 配置允许访问的部门树
- 调整办公室的部门联动策略

### 6.4 首页人类工位放置

支持能力：

- 展示部门内可放置成员
- 若成员不在办公室，自动加入办公室
- 成功后直接落到拓扑图中

## 7. 数据库设计说明

本次数据库变更严格遵循项目规则：

- 所有结构变更都通过 Alembic 迁移文件管理
- 迁移文件统一放在 `nodeskclaw-backend/alembic/versions/`
- 删除一律使用软删除
- 唯一约束使用 Partial Unique Index

本次新增内容包括：

- `departments` 表
- `department_memberships` 表
- `workspaces` 表新增部门范围字段

## 8. 已完成的验证

已经完成的实际验证包括：

- 可以创建树形部门
- 可以为成员分配主部门和兼任部门
- 可以把部门成员设置为部门管理员
- 部门管理员不能创建部门
- 部门管理员不能修改部门外成员归属
- 部门管理员查看组织成员时，只能看到自己部门树内成员
- 部门管理员搜索办公室可加成员时，只能看到自己部门树内成员
- 部门管理员不能把部门外成员加入办公室
- 普通成员可以被加入办公室并成功放置为人类工位

## 9. 当前已知限制

当前还有一个已验证出的限制需要后续继续收口：

- 办公室部门范围对“部门管理员”已经生效
- 但对“组织管理员”目前还没有完全收口

也就是说，当前组织管理员仍然可能看到或加入部门范围外成员。这个问题已经定位到办公室成员过滤路径，需要继续把组织管理员也纳入办公室部门范围约束。

## 10. 适用价值

这套功能对 toB 项目有几个直接价值：

- 支持多企业共用同一套平台
- 支持企业内复杂部门结构管理
- 支持把“谁能管理谁”从组织级进一步细化到部门级
- 支持办公室按业务部门做隔离协作
- 支持普通成员快速进入办公室并参与协同
