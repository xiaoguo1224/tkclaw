# 后端架构设计

## 技术栈

- Python 3.12 + FastAPI + SQLAlchemy（async） + PostgreSQL（火山云 RDS）
- 包管理：uv
- 部署：K8s（火山云 VKE）

## 核心数据模型

### workspace_tasks（工作区任务）

| 列名 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(36) PK | UUID |
| workspace_id | VARCHAR(36) FK | 所属工作区 |
| title | VARCHAR(256) | 任务标题 |
| description | TEXT | 任务描述 |
| status | VARCHAR(20) | pending / in_progress / done / blocked / archived |
| priority | VARCHAR(16) | low / medium / high / urgent |
| assignee_instance_id | VARCHAR(36) FK | 负责 Agent 实例 |
| created_by_instance_id | VARCHAR(36) FK | 创建者 Agent 实例 |
| estimated_value | DOUBLE PRECISION | 预估价值（元） |
| actual_value | DOUBLE PRECISION | 实际价值（元） |
| token_cost | INTEGER | 消耗 token 数 |
| blocker_reason | TEXT | 阻塞原因 |
| completed_at | TIMESTAMPTZ | 完成时间 |
| archived_at | TIMESTAMPTZ | 归档时间 |
| created_at | TIMESTAMPTZ | 创建时间 |
| updated_at | TIMESTAMPTZ | 更新时间 |
| deleted_at | TIMESTAMPTZ | 软删除时间 |

**状态流转**：
- pending -> in_progress（Agent 接取）
- in_progress -> done / blocked
- blocked -> pending / in_progress
- done -> archived（人类管理者手动归档）
- 任何状态 -> archived

### workspace_objectives（工作区目标）

| 列名 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(36) PK | UUID |
| workspace_id | VARCHAR(36) FK | 所属工作区 |
| title | VARCHAR(256) | 目标标题 |
| description | TEXT | 目标描述 |
| progress | DOUBLE PRECISION | 进度 0.0 ~ 1.0 |
| obj_type | VARCHAR(20) | objective（O） / key_result（KR） |
| parent_id | VARCHAR(36) FK | 父级 Objective ID（KR -> O 关联） |
| created_by | VARCHAR(36) FK | 创建者用户 |
| created_at | TIMESTAMPTZ | 创建时间 |
| updated_at | TIMESTAMPTZ | 更新时间 |
| deleted_at | TIMESTAMPTZ | 软删除时间 |

**层级关系**：
- `obj_type=objective`：顶层目标（O），无 parent_id
- `obj_type=key_result`：关键结果（KR），parent_id 指向所属 O

### 认证依赖说明

后端提供两套认证依赖，根据端点是否需要 AI 员工访问来选择：

| 依赖函数 | 认证方式 | 适用场景 |
|---------|---------|---------|
| `get_current_user` | JWT only | 仅人类用户可访问的端点（走廊编辑、组织管理等） |
| `get_current_user_or_agent` | JWT + proxy_token fallback | 人类和 AI 员工均可访问（黑板、成员列表、效能等） |
| `get_current_org` | JWT only + org 解析 | 需要组织上下文的人类专用端点 |
| `get_current_org_or_agent` | JWT / proxy_token + org 解析 | 需要组织上下文且 AI 员工可访问（拓扑、信任策略等） |

`proxy_token` 认证流程：Bearer token 匹配 `instances.proxy_token` -> 找到实例创建者 -> 返回创建者的 User 对象。AI 员工以创建者身份调用 API，继承其组织和工作区权限。

### 持续工作循环机制

- **定时保底**：`WorkspaceSchedule` cron 定时器，每 4 小时发送"任务巡检"系统消息给 Agent
- **事件即时**：任务状态变更时 SSE 广播 `task:status_changed`，后端同时发送系统消息提醒 Agent
- **人类手动**：管理者在黑板创建/分配任务

### 技能学习通知链路

- **触发点**：`handle_learning_callback()`、`_direct_install()`、`install_gene_prerestart()` 在技能学习/安装成功后触发工作区通知
- **受众计算**：以当前 Agent 的 `hex_position_q` / `hex_position_r` 为起点，调用 `get_reachable_endpoints()` 只找 topology（拓扑）里可达的 human（人类成员）和 agent（AI 员工）
- **human 标识处理**：`get_reachable_endpoints()` 返回的 human `entity_id` 实际是 `HumanHex.id`，后端需先用它调用 `deliver_to_human()`，再映射成 `HumanHex.user_id` 生成 `audience_user_ids`
- **输出通道**：
  - `deliver_to_human()`：向可达人类递送“某 Agent 学会了什么”的告知消息，优先走飞书，失败时回退 SSE（服务端推送）
  - `send_system_message_to_agents()`：向可达 Agent 注入系统消息，只做信息同步，不做任务指令
  - `broadcast_event(workspace_id, "agent:skill_learned", ...)`：给 Portal 发送结构化 SSE 事件，载荷包含 `instance_id`、`agent_name`、`gene_name`、`gene_slug`、`summary`、`audience_user_ids`；多工作区场景下，通过 `WorkspaceAgent` 表获取实例所属全部 workspace_id，逐一向各工作区广播
- **前端消费**：Portal 在 `workspace.ts` 订阅 `agent:skill_learned`，`WorkspaceView.vue` 只在当前登录用户 ID 命中 `audience_user_ids` 时显示 toast（提示）
- **逐条播报**：`apply_genome()` 仍按现有逻辑逐个调用 `install_gene()`，因此每个技能学习成功后各自发送一次通知

### Docker Compute Provider 数据目录

Docker 实例数据统一存放在 `DOCKER_DATA_DIR` 指向的目录（默认 `~/.nodeskclaw/docker-instances`），由 `docker_constants.py` 定义。

Docker Compose 部署时，`docker-compose.yml` 中的卷挂载路径可通过 `NODESKCLAW_DATA_DIR` 环境变量覆盖（主要面向 Windows 用户，因 `$HOME` 在 Windows 上不可靠）。容器内的 `DOCKER_DATA_DIR` 环境变量会被设为与 `NODESKCLAW_DATA_DIR` 相同的值，确保后端在容器内生成 per-instance compose 文件时引用的路径是宿主机有效路径。

| 变量 | 作用域 | 说明 |
|---|---|---|
| `DOCKER_DATA_DIR` | 后端进程 | 实例数据存储根目录，来自环境变量或默认 `~/.nodeskclaw/docker-instances` |
| `NODESKCLAW_DATA_DIR` | docker-compose.yml | 卷挂载路径覆盖，Mac/Linux 无需设置，Windows 必填 |

### 实例访问令牌查看与重设

#### Portal（用户门户）接口

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /instances/{instance_id} | 返回实例详情，`env_vars.OPENCLAW_GATEWAY_TOKEN` 用于前端展示脱敏后的访问令牌 |
| POST | /instances/{instance_id}/sync-token | 从运行中 Pod 日志同步当前令牌，仅用于补齐缺失令牌 |
| POST | /instances/{instance_id}/regenerate-token | 生成新的访问令牌，写回配置并触发实例重启 |

#### 数据一致性要求

- **数据库字段**：
  - `instances.proxy_token`：作为 OpenClaw gateway（网关）认证令牌来源
  - `instances.env_vars.OPENCLAW_GATEWAY_TOKEN`：实例环境变量中的访问令牌
  - `instances.env_vars.NODESKCLAW_TOKEN`：与 `OPENCLAW_GATEWAY_TOKEN` 保持一致，供实例侧 Agent 通信使用
- **运行时配置**：
  - ConfigMap 中的 `OPENCLAW_GATEWAY_TOKEN` 与 `NODESKCLAW_TOKEN` 必须与数据库保持一致
  - `openclaw.json` 中的 `gateway.auth.token` 由 `instance.proxy_token` 注入

#### 重设链路

1. Portal 调用 `POST /instances/{instance_id}/regenerate-token`
2. 后端生成新令牌（沿用 `secrets.token_hex(24)`）
3. 后端同步更新 `env_vars.OPENCLAW_GATEWAY_TOKEN`、`env_vars.NODESKCLAW_TOKEN`、`instances.proxy_token`
4. 后端替换实例对应 ConfigMap
5. 后端复用 `restart_instance()` 触发实例重启
6. Portal 轮询实例状态，待 `restarting`（重启中）恢复为 `running`（运行中）后结束提示

#### 失败处理

- ConfigMap 更新或重启触发失败时，后端需回滚数据库中的令牌相关字段
- 如已写入 ConfigMap，则需尽量恢复旧 ConfigMap 内容，避免前端显示新令牌、实例仍运行旧令牌的状态分裂
- `sync-token` 在补齐令牌时，也必须同步回填 `instances.proxy_token`，避免 `gateway.auth.token` 继续读取旧值或空值
