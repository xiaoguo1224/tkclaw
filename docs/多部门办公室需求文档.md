# 多部门办公室需求文档（V1.2）

## 1. 文档目标

本文档用于定义“多组织场景下，部门与办公室联动管理”的产品需求，作为后续功能说明、实现设计、联调测试的统一依据。

本文档聚焦两类核心诉求：

- 部门与办公室采用多对多关联（一个部门可关联多个办公室，一个办公室可关联多个部门）
- 部门支持负责人配置与维护（可设置/移除部门负责人）

## 2. 业务背景

当前系统已具备多组织、多部门基础能力，但在业务使用中存在以下问题：

- 办公室与部门关系表达不够清晰，缺少标准化多对多归属模型
- 用户无法在部门管理页面直接指定负责人，管理链路不闭环
- 办公室列表缺少按部门查看能力，不利于运营和管理

toB 场景中，一个企业内通常存在“多个部门共用一个办公室”与“一个部门使用多个办公室”的情况，因此需要明确支持多对多模型。

## 3. 术语定义

- Organization（组织）：企业级隔离边界
- Department（部门）：组织内树形管理单元
- Workspace（办公室）：协作空间
- Department Manager（部门负责人）：在部门成员关系中 `role = manager` 的成员
- Workspace-Department Link（办公室部门关联）：部门与办公室关系记录

## 4. 角色与权限

### 4.1 角色

- 平台超管：跨组织管理
- 组织管理员：管理本组织全部部门和办公室
- 部门负责人：管理本部门树成员及受控办公室成员操作
- 组织成员：普通成员

### 4.2 权限规则

- 组织管理员可维护任意部门与办公室关联关系
- 部门负责人只能在其部门树范围内进行成员管理
- 部门负责人不能创建/删除部门
- 部门负责人不能越权管理部门外成员归属
- 办公室成员权限（`WorkspaceMember.permissions`）继续生效，不被部门角色替代

## 5. 需求范围

### 5.1 本期范围（In Scope）

- 部门与办公室多对多关系管理
- 办公室列表按部门筛选查看
- 办公室设置中维护关联部门
- 部门页面维护负责人（设置/移除）
- 前后端接口与数据结构同步
- i18n 文案同步

### 5.2 非本期范围（Out of Scope）

- 部门跨组织共享
- 矩阵组织（多重汇报线）高级流程
- 自动审批流
- 历史操作回滚 UI

## 6. 功能需求

### FR-01 部门与办公室多对多关联

- 系统必须支持 Department 与 Workspace 多对多关系
- 一个办公室可关联多个部门
- 一个部门可关联多个办公室
- 关系删除采用软删除

验收标准：

- 关联后可在数据库查询到有效关系记录
- 同一 `workspace_id + department_id` 不可重复有效写入
- 取消关联后仅做逻辑删除，不做物理删除

### FR-02 办公室设置维护关联部门

- 在办公室设置页面可选择并保存关联部门列表
- 支持多选部门
- 保存后办公室详情返回最新关联部门列表

验收标准：

- 保存成功后刷新页面，关联部门状态一致
- 清空关联后，办公室可回到组织级可见逻辑（由策略字段控制）

### FR-03 办公室列表按部门筛选

- 办公室列表页提供“按部门查看”筛选器
- 选择部门后，仅显示与该部门（含其子部门范围，如适用）关联的办公室
- 未选择部门时，展示默认可见办公室列表
- 已实现规则：在选择部门时，部门负责人可选择“管理部门树（含子部门）+ 自己参与办公室所属部门”；普通员工仅可选择“自己参与办公室所属部门”；组织管理员可选择全组织部门
- 已实现防越权：`GET /workspaces` 传入超出可选范围的 `department_id` 时返回 403

验收标准：

- 选择已关联部门时，能看到对应办公室
- 选择未关联部门时，列表为空或仅显示策略允许的办公室
- 切换筛选项后结果实时更新

### FR-04 部门负责人配置

- 部门管理页可查看当前负责人列表
- 可将部门成员设为负责人
- 可移除负责人身份（降级为普通成员）

验收标准：

- 设置负责人后，成员显示负责人身份
- 移除负责人后身份即时更新
- 同一成员可在不同部门担任负责人（若业务允许）

### FR-05 部门负责人边界控制

- 部门负责人执行成员管理操作时，必须限制在本部门树范围
- 对部门外成员操作应被拒绝并返回明确错误

验收标准：

- 对部门外成员执行“调整归属/加入办公室”等动作，返回权限错误
- 对部门内成员执行同类动作，允许成功

### FR-06 人类工位联动

- 放置人类工位时，候选成员来源需要与部门范围规则一致
- 当候选成员未加入办公室时，系统可按既有流程先加入办公室再放置

验收标准：

- 部门内成员可被正常放置
- 部门外成员不可通过越权路径放置

### FR-07 办公室与部门联合显示
- 已实现：办公室列表卡片展示主管部门（展示该办公室关联的全部部门）与成员数
  
验收标准：
- 主页面能看到办公室所属主管部门（多部门）以及成员数

## 7. 数据需求

### 7.1 数据模型要求

- 新增办公室-部门关联实体（例如 `workspace_departments`）
- 关键字段：
  - `workspace_id`
  - `department_id`
  - `org_id`
  - `deleted_at`
  - `created_at`
  - `updated_at`

### 7.2 数据规则

- 所有删除均为软删除
- 关系唯一约束使用 Partial Unique Index（`deleted_at IS NULL`）
- 所有结构变更必须提供 Alembic 迁移文件，放置于 `nodeskclaw-backend/alembic/versions/`

## 8. 接口需求

### 8.1 办公室列表

- `GET /api/v1/workspaces`
- 新增可选参数：`department_id`
- 按部门筛选返回办公室列表
- 返回项增加：
  - `department_names: string[]`
  - `member_count: number`

### 8.1.1 办公室筛选部门选项

- `GET /api/v1/workspaces/filter-departments`
- 返回当前用户可用于办公室列表筛选的部门集合（按权限动态计算）

### 8.2 办公室设置

- `PUT /api/v1/workspaces/{workspace_id}`
- 支持保存关联部门列表（沿用或映射到多对多关系）

### 8.3 部门负责人管理

- 复用部门成员接口：
  - `GET /api/v1/orgs/{org_id}/departments/{department_id}/members`
  - `POST /api/v1/orgs/{org_id}/departments/{department_id}/members`
  - `PUT /api/v1/orgs/{org_id}/departments/{department_id}/members/{membership_id}`
  - `DELETE /api/v1/orgs/{org_id}/departments/{department_id}/members/{membership_id}`

负责人语义：

- 设置负责人：`role = manager`
- 取消负责人：`role = member`

## 9. 页面需求

### 9.1 办公室列表页

- 新增部门筛选下拉
- 支持“全部部门”选项
- 筛选项由后端接口动态返回，不再直接使用组织全量部门树
- 空状态文案根据筛选状态区分
- 办公室卡片展示主管部门（多部门）与成员数

### 9.2 办公室设置页

- 支持多部门关联选择
- 保存后即时反馈成功/失败

### 9.3 部门管理页

- 新增“管理负责人”入口
- 弹窗展示当前负责人
- 支持设置/移除负责人

## 10. 非功能需求

- 兼容 CE/EE 现有架构
- 不破坏既有组织/办公室数据
- 关键接口响应时间保持在现有可接受范围
- 所有新增用户可见文案必须 i18n 化

## 11. 验收用例（最小集合）

1. 创建部门 A、B，创建办公室 X、Y
2. 将 A 关联 X、Y；将 B 关联 X
3. 部门筛选 A 时应看到 X、Y；筛选 B 时应看到 X
4. 将成员 M 设为 A 部门负责人，确认身份显示正确
5. M 对 A 内成员操作成功，对 B-only 成员操作失败
6. 放置人类工位时，候选成员与部门规则一致

## 12. 风险与注意事项

- 历史字段兼容：若存在旧字段（如数组形式部门列表），需保证平滑迁移与回填
- 权限一致性：组织管理员、部门负责人、办公室管理员三者权限边界必须统一
- 同源逻辑：成员搜索、办公室成员添加、工位放置候选来源需同步规则，避免前后不一致
