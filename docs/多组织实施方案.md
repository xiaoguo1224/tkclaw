# 多组织树形部门与权限管控实施方案（含数据库迁移规范）

## Summary
在现有“多组织 + 组织成员 + 办公室成员权限”基础上，新增树形部门体系，支撑 toB 多公司、多部门、分级管理场景。  
V1 采用“树形部门 + 主部门/兼任部门 + 办公室联动 + 组织/部门分层权限”模型，并严格遵循 [AGENTS.md](/f:/A____usually/tk/tkclaw/AGENTS.md) 的数据库规则：所有结构变更必须通过 Alembic 迁移文件管理（`nodeskclaw-backend/alembic/versions/`），删除一律软删除，唯一约束使用 Partial Unique Index。

## Key Changes
### 领域模型
- 新增 `Department`（部门）模型：
  - 归属 `org_id`
  - `parent_id` 支持树形结构
  - 包含 `name`、`slug/code`、`description`、`sort_order`、`is_active`
  - 继承现有软删除基类
- 新增 `DepartmentMembership`（部门成员关系）模型：
  - `user_id` + `department_id`
  - `role`：`member` / `manager`
  - `is_primary`：主部门标记
  - 继承现有软删除基类
- 办公室增加部门联动配置：
  - `visibility_scope`
  - `allowed_department_ids`
  - `auto_sync_mode`
- 不改现有 `OrgMembership` 的组织级准入职责
- 不移除现有 `WorkspaceMember` 的办公室细粒度权限职责

### 权限模型
- 平台超管：跨组织全局管理，保持现状
- 组织管理员：管理整个组织、所有部门树、所有办公室及成员
- 部门管理员：仅管理“本部门及子部门”成员归属、候选成员、关联办公室范围
- 组织普通成员 / operator：不自动拥有部门管理权限
- 办公室权限继续以 `WorkspaceMember.permissions` 为准
- 部门只决定“管理范围、可见范围、可加入范围”，不直接替代办公室权限

### 办公室联动
- 办公室默认保持“全组织可见”，兼容老逻辑
- 开启部门范围后：
  - 只能搜索允许部门树内成员
  - 只能把允许部门树内成员加入办公室
  - 首页“放置人类工位”只展示允许范围内成员
- 选中尚未加入办公室但属于允许部门树的成员时：
  - 先加入办公室
  - 再执行创建人类工位/后续动作
- 办公室成员展示补充成员主部门/兼任部门信息

## Database Changes
### 数据库规则
- 所有数据库结构变更必须同步创建 Alembic 迁移文件
- 迁移文件必须放在 `nodeskclaw-backend/alembic/versions/`
- 不允许只改 ORM Model，不补迁移文件
- 删除语义一律软删除，不允许物理删除

### 建表与索引
- `departments`
  - 按组织隔离
  - `slug/code` 在同一组织内唯一
  - 唯一索引必须使用 `deleted_at IS NULL`
- `department_memberships`
  - 用户和部门关系唯一
  - 用户在同一组织仅允许一个主部门
  - 所有唯一约束必须用 Partial Unique Index
- 若办公室部门范围落表而非 JSON 字段：
  - 单独新增关联表并加软删除字段
  - 关联唯一约束也使用 Partial Unique Index

### 迁移拆分建议
- 按“单一变更单元”拆分 Alembic revision
- 结构迁移与数据回填可拆分为两个 revision，避免回滚复杂度升高
- 每个 revision 需要可独立 `upgrade`/`downgrade`

## Public APIs / Interfaces
### 后端接口
- 新增部门管理接口：
  - `GET /orgs/{org_id}/departments`
  - `POST /orgs/{org_id}/departments`
  - `PUT /orgs/{org_id}/departments/{department_id}`
  - `DELETE /orgs/{org_id}/departments/{department_id}`
- 新增部门成员接口：
  - `GET /orgs/{org_id}/departments/{department_id}/members`
  - `POST /orgs/{org_id}/departments/{department_id}/members`
  - `PUT /orgs/{org_id}/departments/{department_id}/members/{membership_id}`
  - `DELETE /orgs/{org_id}/departments/{department_id}/members/{membership_id}`
- 组织成员接口扩展：
  - 成员返回值补充主部门、兼任部门、是否部门管理员
  - 直接创建成员/邀请成员支持指定部门归属
- 办公室接口扩展：
  - 办公室详情/设置增加部门范围配置
  - 组织成员搜索、办公室成员添加、首页候选成员查询统一接入部门过滤

### 前端页面
- Portal 新增部门管理页
- 组织成员页增加部门筛选、部门归属编辑
- 办公室设置页增加部门范围配置
- 首页“放置人类工位”与办公室加人弹窗统一接入部门范围逻辑
- 所有新增文案同步补齐中英文 i18n

## Implementation Changes
### 后端
- 新增 model、schema、service、API、权限检查逻辑
- 为部门树提供统一查询 service，避免各处重复递归
- 所有查询统一补 `not_deleted(...)`
- 所有删除动作统一改为 `soft_delete()`
- 新增审计日志：
  - `department.created`
  - `department.updated`
  - `department.deleted`
  - `department.member_added`
  - `department.member_removed`
  - `workspace.department_scope_updated`

### 前端
- 组织页增加树形部门管理能力
- 组织成员页支持维护主部门/兼任部门
- 办公室设置页支持配置允许部门树
- 首页与办公室成员相关交互统一改造成“组织成员 + 部门范围 + 办公室范围”联合过滤
- 如 EE 管理前端存在同源成员/办公室逻辑，必须同步修改

## Test Plan
- 多组织隔离：
  - A 组织看不到 B 组织的部门、成员、办公室
- 树形部门：
  - 创建父子部门成功
  - 删除存在子部门或成员的部门时按规则拒绝
- 成员归属：
  - 成员仅有一个主部门
  - 可添加多个兼任部门
  - 软删除后可重新建立关系
- 权限边界：
  - 组织管理员可管理全组织
  - 部门管理员仅管理本部门树
  - 普通成员无法越权调整部门或办公室范围
- 办公室联动：
  - 限定部门后，只能搜索/添加允许部门树成员
  - 首页放置人类工位支持允许范围内普通成员
  - 不在允许范围内的成员无法加入或放置
- 兼容性：
  - 老办公室未配置部门范围时保持现有行为
  - 老成员未分配部门时，组织管理员仍可见并补录
- 数据库验证：
  - Alembic migration 可执行（`alembic upgrade head`）
  - Partial Unique Index 生效
  - 软删除记录不会阻塞新建同名/同关系数据

## Assumptions
- V1 采用树形部门，不做跨组织部门或矩阵汇报
- 一个成员必须先是组织成员，才能加入该组织的部门
- 一个成员必须且仅有一个主部门；兼任部门可多个
- 部门负责人通过 `DepartmentMembership.role = manager` 表达
- 办公室权限保留现有模型，部门不直接覆盖办公室细粒度权限
- 本次交付中，代码改动与对应 Alembic 迁移必须作为同一个提交单元完成
